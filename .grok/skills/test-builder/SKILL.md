name: test-builder

[任务]
    为当前项目建立并执行**务实回归测试**：探测/搭建测试基建 → 为高价值逻辑写测试 → 跑测试 → 输出结构化报告。
    让 dev-builder「四步走验证」第2步「测试完整性」从口号变成真卡点（跑套件、贴证据），并为关键逻辑留下抵御回归的防线。
    **测试代码的编写遵循「主 Agent 不亲自写代码」铁律**：派 tester Sub-Agent（或非该功能作者的另一 implementer fresh 实例）写，主 Agent 写提示词 + 验收。测试代码如需再审查，走 code-reviewer Sub-Agent。

[依赖检测]
    Skill 启动时第一步自动执行：

    必需：
    - 项目代码已存在 → 无代码则提示先调用 /dev-builder
    - 测试运行器（按技术栈）：前端 vitest / 后端 pytest → 缺失则按「基建探测与搭建」自主 scaffold

    可选（增强能力）：
    - Product-Spec.md → 有则据「自反一致性契约」「数据规则」等条目定位必测项
    - DEV-PLAN.md → 有则对照当前 Phase/Task 范围定测试边界
    - git → 有则用 git diff 锁定本次改动、优先覆盖新增/改动逻辑

    安装策略：
    - 测试框架缺失时，Agent 自主判断并安装（pytest 进 api/requirements*.txt 或 venv；vitest 进 web devDependencies），最小化配置
    - 需要用户权限/交互时提示用户操作
    - 可选依赖缺失时标记降级，不阻塞

[第一性原则]
    **契约优先**：测试预算先砸在「出错最贵」的地方——跨边界契约（如 KMZ 导出↔导入自反一致性）、解析器、去重主键、状态迁移。这些破了是数据级灾难，比 UI 像素重要得多。
    **真跑真证据**：说"测过了"必须附测试运行器的真实输出（passed/failed 计数 + 用例名）。没有运行输出的"通过"等于没测。
    **不追覆盖率虚荣**：不以覆盖率数字为目标，不为凑数写断言空洞的测试。一个能挡住真实回归的用例，胜过十个测 getter 的用例。
    **测试是回归防线、不是临时验证**：留下来的必须可重跑、可入库、下次改动自动挡。一次性 `python -c` 验证不算交付。
    **写测与被测分离**：测试编写者**不得**是被测代码的作者——自码自测会把同一套（可能错的）假设也写进断言，测试"绿"只证明代码符合作者想象、不符合 Spec（confirmation bias）。测试必须交独立方写：优先 tester Sub-Agent，或非该功能作者的另一 implementer fresh 实例。
    **联网优先**：涉及不熟悉的测试框架用法、fixture 模式、mock 数据库时先 WebSearch 确认最佳实践再动手。

[输出风格]
    **语态**：
    - 像务实的测试工程师：先问"这块破了代价多大"，再决定测不测、怎么测
    - 每个结论附测试运行器真实输出，不空口

    **原则**：
    - × 绝不说"应该没问题"、"大概能过"——跑一遍拿计数
    - × 绝不为覆盖率写无意义用例
    - × 绝不把一次性验证脚本当成交付的测试
    - ✓ 明说测了什么、没测什么、为什么这么取舍
    - ✓ 失败用例贴：用例名 + 期望 vs 实际 + 文件:行号
    - ✓ 新增测试列清单，说清各自防的是哪类回归

    **典型表达**：
    - "自反一致性契约是命脉——加 test_roundtrip：导出含 NP 点 KMZ → 重导入，断言 site/road/lessor 计数回灌且范围圈 0 泄漏。pytest 5 passed。"
    - "解析器表头探测（#35）只在主路径验过，补 3 个用例覆盖旧/新模板 + 缺 SITE ID 报错。"
    - "UI 像素、三方库行为不测（性价比低）；本轮聚焦后端契约 + 解析。前端只测 utils 纯函数（metersToProjRadius 等）。"

[测试维度清单]
    判断"测什么"——按价值排序，不是按代码行平铺。

    必须测（破了是数据级/契约级灾难）：
    - **跨边界契约**：导出↔导入往返一致性、序列化↔反序列化、API 请求↔响应契约
    - **解析器 / 数据清洗**：多源格式解析、字段名规范化、表头探测、异常输入容错
    - **去重 / 合并主键**：去重键计算、冲突判定、覆盖/忽略决策
    - **关键边界与异常**：空值/null、nan/inf、越界坐标、超大/负数、坏编码——尤其防御性 return 分支

    推荐测：
    - 纯函数工具（坐标换算、格式化、状态映射等无副作用逻辑）
    - 状态迁移 / 枚举映射（如状态值规范化、类别→站型映射）
    - 关键 API endpoint 的入参校验与回落（白名单、缺省值）

    可选测：
    - 复杂查询的 SQL 结果形状（需测试库时）
    - 有 Playwright 时的核心交互流程 E2E

    明确不测（性价比低，写了也别写）：
    - UI 像素 / 快照（脆、维护重，靠人工眼检 + 设计稿对照）
    - 三方库自身行为（信任其已测）
    - getter/setter、纯透传、无分支的样板代码

[测试策略]
    [基建探测与搭建]
        1. 探测技术栈与现有测试：找 pytest/vitest 配置、test_*.py / *.test.ts、tests 目录、CI。
        2. 无基建则**最小化 scaffold**（参照 templates/test-scaffold.md）：
           - 后端：pytest（+ 必要时 pytest-asyncio）；约定 `api/tests/`；一个能跑通的样例用例验证基建可用
           - 前端：vitest；约定就近 `*.test.ts` 或 `web/src/__tests__/`；一个样例验证基建可用
        3. scaffold 本身也派子 Agent 落地，主 Agent 验"能跑空套件"再继续。

    [取舍策略]
        - 先列候选（对照[测试维度清单] + 本次 git diff / 当前 Task 范围），按"破了代价"排序。
        - 只写必须测 + 高价值推荐测；可选测按预算和用户意图决定。
        - 每个取舍**说明理由**，不闷头写也不闷头跳。
        - 数据库依赖：优先用内存/构造数据（如直接喂 dict 行给纯函数/构造器），避免起真库；必须真库时用最小 fixture + 测试库，绝不碰生产/基线数据。

    [编写与执行（角色路由）]
        - **铁律：写测与被测分离**——测试编写者不得是被测代码的作者（见[第一性原则]）。
          路由优先级：① 被测代码刚由 implementer 写的 → 测试派 **tester Sub-Agent** 写；② 或派**非该功能作者的另一 implementer fresh 实例**；③ 主 Agent 始终不亲自写测试代码。
        - 主 Agent 写测试提示词（要测什么、断言什么、用例清单、放哪），交独立方落地。
        - 独立方写完跑测试，回报运行器原始输出。
        - 主 Agent **独立复核**：自己拉真实输出验证（可自己跑测试命令——属验收，不是写产品代码），不轻信其自述。
        - 测试代码本身若再要审查，同样找**未参与写该测试**的一方。

    [失败处理闭环]
        - 测试**红**（失败）分两种：
          - 测试发现真 bug → 调 bug-fixer 修业务代码 → 重跑
          - 测试本身写错（断言/夹具错）→ 派 tester 修测试 → 重跑
        - 判断是"代码错"还是"测试错"前，先看失败信息和 Spec 契约，别条件反射改测试迁就代码。
        - 循环直到全绿，贴最终运行输出。

[测试完成度判断]
    本轮测试视为"完成"的门槛（对照 dev-builder 四步走第2步）：

    必须满足：
    - 测试基建可用（套件能跑、有真实运行输出）
    - [测试维度清单] 的「必须测」项中，**与本次开发/改动相关的**已覆盖并全绿
    - 失败用例已闭环（要么修代码、要么修测试），无遗留红
    - 新增用例已入库（可重跑），不是一次性脚本

    输出证据（汇报时必须附）：
    - 测试运行器原始输出（passed/failed 计数）
    - 本轮新增/修改的用例清单 + 各自防的回归类型
    - 明确列出"未测项 + 不测理由"，不假装全覆盖

    **时效性**：汇报测试结果的命令必须在同一消息中执行；若其后又改了代码，重跑。不接受"前面跑过了"。

[工作流程]
    [第一步：依赖检测 + 基建探测]
        执行 [依赖检测] + [测试策略] 的「基建探测与搭建」。
        无基建 → 派子 Agent scaffold → 主 Agent 验"空套件能跑"。

    [第二步：圈定测试范围]
        - 读 Product-Spec.md 的契约/数据规则条目 + DEV-PLAN.md 当前 Phase/Task + git diff。
        - 对照 [测试维度清单] 列候选，按 [取舍策略] 排序取舍，产出"本轮测试清单 + 取舍理由"。

    [第三步：写测 + 跑测]
        - 主 Agent 写测试提示词，派 tester（或非作者的另一 implementer 实例）落地（按 [编写与执行] 路由）。
        - 跑测试，主 Agent 独立复核运行输出。

    [第四步：失败闭环]
        - 有红 → 按 [失败处理闭环] 区分代码错/测试错 → 修 → 重跑，直到全绿。

    [第五步：完成度判断 + 汇报]
        - 遵守 AGENTS.md「完成声明五步闸」：先新跑测试命令并读完整输出，再说"测试通过"。
        - 对照 [测试完成度判断] 逐项确认，附运行输出 + 新增用例清单 + 未测项理由。
        - 嵌入 review→fix 闭环时：作为 per-Task「Stage 1 规格通过后」的测试关，或 Phase 四步走第2步的真验证；不通过不放行。

[初始化]
    执行 [第一步：依赖检测 + 基建探测]
